# struct-doc-answer — 知识文档生成

## ⚠️ 强制检查清单

| # | 检查项 | 合格标准 |
|---|--------|----------|
| 1 | **TodoWrite 开任务** | 收到 URL 后立即开 TodoWrite，第一项永远是查重 |
| 2 | **查重 flomo_memo_search** | **唯一可靠的查重方式**——创建本地文档前必须调用 MCP 工具 `flomo_memo_search`，搜主题关键词 + 子领域，5 条以上结果需逐条评估是否重复 |
| 3 | 收到 URL → **立即 webfetch** | 不得以"之前处理过"为由跳过 |
| 4 | 已读完整内容 | 不仅是标题/摘要/URL，必须读完全文 |
| 5 | 文件名格式 | `领域_二级领域_知识点.md`，三段式，每段≥2字符 |
| 6 | 标签数量 | ≥3个，必含 `#信号类型`，咨询类必含 `@咨询线` |
| 7 | 本地 = flomo 格式 | 直接写 flomo 格式，**无需**转换 |
| 8 | flomo 上传 | 调用 `flomo_memo_create` 上传（直接传本地文档内容） |
| 9 | 无非 flomo 语法 | 禁止 `#` 标题/引用/代码块/链接/图片/水平线/表格 |
| 10 | 标签在第一行 | 第一行必须是 `#xxx` 或 `@xxx` 标签 |
| 11 | **answers 严禁推送到远程** | 上传 flomo 后必须 `git reset HEAD~1` 清除本地 commit，**永远不要 push answers/ 内容** |

> **核心红线：禁止跳过。每条输入必须产出文档。禁止使用 Markdown 标题等非 flomo 语法。**
>
> **查重 = `flomo_memo_search`，不是别的**：本地 hook 的相似度检测只是辅助手段，**唯一权威查重是 `flomo_memo_search`**——因为 flomo 才是真实笔记库，git 历史和 working tree 都不完整。漏掉 flomo 查重会导致上传已有内容的重复笔记，浪费 flomo 配额、增加后续检索成本。
>
> **answers 目录严格本地化**：所有 `answers/` 下的文档都是为 hook 验证临时创建的本地草稿，**永远不要 push 到 origin**。git remote 仓库只保留 `hooks/`、`scripts/`、`SKILL.md` 等基础设施代码。

## 一、标题与路径规则

### 1.1 文件名格式

**格式：** `领域_二级领域_知识点.md`（名词，三段式）

| 示例 | 说明 |
|------|---------|
| `科技_AI芯片_专用处理器设计范式.md` | 文件名即文档标题 |
| `金融_股票策略_估值模型分析.md` | 同上 |

**知识点三要求：**
- 可独立识别：能用一个名词短语指代一个具体的事件、对象、技术、现象或方法
- 可作为文档主语：文档全部内容围绕这个知识点展开
- 可挂载多个事实：可容纳多个数据点、案例、机制描述

---

## 二、文档结构（= flomo 格式）

**本地文档就是 flomo 格式**，无需任何转换。第一行标签，`**加粗**`代替标题。

```markdown
#信号笔记 #技术 #AI大模型

**科技_AI大模型_Grok4.5开放**

来源：来源1、来源2

**第一段加粗要点**：正文内容...
正文继续...

**第二段加粗要点**：正文内容...

- 列表项 1
- 列表项 2
- 列表项 3
```

### 2.1 允许的 flomo 语法

| 语法 | 用途 | 示例 |
|------|------|------|
| `**加粗**` | 段落/段内强调 | `**核心要点**：xxx` |
| `<mark>高亮</mark>` | 关键术语 | `<mark>智能向善</mark>` |
| `<u>下划线</u>` | 次级强调 | `<u>重要</u>` |
| `- xxx` | 无序列表 | - 案例 1 |
| `1. xxx` | 有序列表 | 1. 第一步 |

### 2.2 禁止的语法

| 语法 | 原因 |
|------|------|
| `#`/`##`/`###` 标题 | flomo 不支持 |
| `>` 引用块 | flomo 不支持 |
| ` ``` ` 代码块 | flomo 不支持 |
| `[标题](url)` 链接 | flomo 不支持 |
| `![图片](url)` 图片 | flomo 不支持 |
| `---` 水平线 | flomo 不支持 |
| `\|` 表格 | flomo 不支持 |

### 2.3 段落结构建议

每个文档通常包含：
1. **标题**（`**领域_二级领域_知识点**`）— 第一个加粗段
2. **来源行**（`来源：xxx、xxx`）
3. **核心要点**（多个 `**加粗**` 段落，每个段落讲一个子主题）
4. **可选列表**（用 `-` 或 `1.` 列举要点）

### 2.4 标签规则

**第一行必须是标签行**：
- ≥3 个标签
- 必含 `#信号类型`（五选一）
- 咨询类必含 `@咨询线`
- 领域标签：`#技术` `#政治` `#经济` 等
- 二级领域标签：`#AI大模型` `#外交` 等
- 具体概念标签：自由添加

**示例：**
- `政治_国际合作_中国AI全球治理担当.md` → `#信号笔记 #政治 #国际合作 #人工智能 #全球治理`
- `经济_农业经济_数字乡村农业全要素生产率增长.md` → `#趋势信号 #经济 #农业经济 #数字乡村 #农业TFP`

## 三、核心操作流程

### 3.1 标准流程（网页/文章）

收到 URL 后**立即开 TodoWrite**，第一项永远是查重。

| 步骤 | 操作 | 校验点 |
|------|------|--------|
| 1 | **TodoWrite 开任务** | 第一项：`flomo_memo_search 查重` |
| 2 | **`flomo_memo_search` 查重（必须）** | 调用 MCP 工具 `flomo_memo_search`，搜主题关键词 + 子领域；**这是唯一权威查重方式** |
| 3 | `webfetch <URL>` | 获取完整内容（必须读完） |
| 4 | 确定文件名 `领域_二级领域_知识点.md` | 文件名格式正确？三级结构？ |
| 5 | 创建本地文档 `answers/领域/二级领域/文件名.md` | flomo 格式？标签在第一行？ |
| 6 | `git add` → hook 验证 | 通过才能继续；hook 检测到同子目录相似标题会**警告** |
| 7 | `flomo_memo_create` 上传 | 直接传本地文档内容 |
| 8 | `git reset --hard HEAD~1` | 清除本地 commit（**不 push**） |

**flomo 查重命中后的处理**：
- 完全同主题 → 在原 flomo 笔记上 `flomo_memo_update` 追加，而非新建
- 主题有重叠但角度不同 → 创建新文档但要明确"补充视角"
- 完全独立 → 正常新建上传

### 3.2 微信公众号流程

1. TodoWrite + `flomo_memo_search` 查重
2. `webfetch` 获取正文
3. 解析 HTML 提取正文文本内容
4. 创建本地文档（flomo 格式）并走标准流程 3.1 第 5-8 步

### 3.3 期刊目录页流程

**必须逐一处理每篇论文，禁止合并。**

1. TodoWrite 为每篇论文开单独任务清单
2. 每篇独立文件名 → 查重 → 创建文档 → hook 验证 → flomo 上传
3. flomo 上传必须逐篇进行，不得合并

### 3.4 sub-agent 委托模式（可选）

对于长篇文章或复杂主题，可将"webfetch + 查重 + 路径建议"打包委托给 sub-agent：

```
委派 sub-agent 的输出应该包括：
1. 文章主题摘要
2. 建议文件名（领域_二级领域_知识点.md）
3. 建议标签（≥3个）
4. 关键概念列表（3-5个）
5. 是否与现有 flomo 笔记重复的判断
```

主流程拿到 sub-agent 输出后，进入步骤 5（创建本地文档）。

## 四、标签体系

### 4.1 咨询服务线标签（@前缀）

咨询类文档强制，至少1个，通常不超过3个

| 标签 | 适用场景 |
|------|---------|
| `@行业研究` | 行业深度分析、市场规模评估、竞争格局梳理 |
| `@战略规划` | 客户战略制定、增长策略、业务组合优化 |
| `@竞争情报` | 竞争对手动态追踪、竞品分析 |
| `@政策预警` | 监管政策变化评估、合规风险识别 |
| `@运营诊断` | 供应链/渠道/效率分析 |
| `@市场进入` | 新市场/新客户进入策略 |
| `@技术尽调` | 技术评估，投资标的分析 |
| `@风险管理` | 地缘政治/供应链/合规风险评估 |
| `@组织人力` | 组织变革、人才策略、薪酬设计 |
| `@资本咨询` | 资本运作、并购、IPO策略 |

### 4.2 信号类型标签（#前缀）

每个文档必须选择一个

| 标签 | 含义 | 特征 | 时效性 |
|------|------|------|--------|
| `#知识基座` | 长期稳定的概念/机制/框架 | 体系化论述，含机制解释和多案例 | 数年有效 |
| `#趋势信号` | 正在发生的结构性变化 | 含趋势判断、方向性分析 | 季度-年度有效 |
| `#信号笔记` | 单次事件/数据点记录 | 以事件为核心，含咨询含义推断 | 周-月有效 |
| `#分析框架` | 可复用的方法论/分析模型 | 非领域知识，是思考工具 | 数年有效 |
| `#知识载体` | 工具/资源/数据集/可视化 | 承载和呈现结构化知识 | 数年有效 |

> 企业/行业/监管信息默认 `#信号笔记`

### 4.3 标签格式规范

**所有标签必须以 `#` 或 `@` 开头，无一例外。**

| 检查项 | 标准 |
|--------|------|
| 数量 | ≥3个 |
| 必含 | `#信号类型`（五选一） |
| 咨询类 | 必含 `@咨询线`（至少1个） |
| 领域 | 文件名中的一级领域，**必须加 `#`**，如 `#技术` `#政治` |
| 二级领域 | 文件名中的二级领域，**必须加 `#`**，如 `#AI大模型` `#外交` |
| 格式 | 标签之间用空格分隔，无其他符号包围 |
| 位置 | **第一行**，紧贴行首 |

**正确示例：**
```
#信号笔记 #技术 #AI大模型
```

**错误示例：**
```
信号笔记 技术 AI大模型  （错误：无 # 前缀）
```

## 五、同一概念判断

| 场景 | 判断 | 操作 |
|------|------|------|
| 核心术语完全相同 | ✅ 同一概念 | 更新补充新角度/案例 |
| 术语不同但互为同义词或上下位关系 | ✅ 同一概念 | 合并到更宽泛或更成熟的文件 |
| 同一事件/政策的不同角度分析 | ✅ 同一概念 | 合并到同一文件的不同段落 |
| 核心术语相同但分析视角完全不同 | ⚠️ 视情况 | 若段落可容纳则合并，否则新建并互链 |
| 不同领域、不同实体、不同分析框架 | ❌ 不同概念 | 新建独立文件 |

## 六、更新规则

### 6.1 更新原则：增量修改，禁止覆盖

**判断标准：**
- 文件名中的核心术语与已有文档完全相同 → 增量更新
- 文件名中的核心术语部分重叠、可以合并 → 增量更新
- 仅来源/案例/数据不同，概念相同 → 增量更新

**正确更新方式：**

| 要更新的内容 | 操作方法 |
|------------|---------|
| 新来源 | 追加到 `来源：` 一行的末尾，用顿号分隔，去重 |
| 新要点/新机制/新技术 | 新增 `**加粗要点**`：正文内容... |
| 新事实/新案例 | 追加到对应要点段落的末尾 |
| 标签补充 | 修改第一行标签列表，新增补充的 |

**禁止全量覆盖：**
- ❌ 禁止用 `write` 直接重写整个文件
- ❌ 禁止删除原文重新创建
- ❌ 禁止用新内容替换旧内容的方式"更新"

## 七、咨询视角信号分析法

### 第一层：存在性追问
- 为什么是这个时间点出现？
- 为什么是这个公司/这个人？
- 为什么这个故事被报道？（主动释放信号/被动回应猜测/行业吹风）

### 第二层：咨询含义推断
- 这篇文章对什么类型的咨询问题有用？
- 这是趋势信号还是噪音？
- 是否需要与其他文档交叉印证？

### 第三层：文档类型选择
- `#知识基座` → 创建/更新机制文档
- `#趋势信号` → 创建/更新趋势判断文档
- `#信号笔记` → 创建事件记录文档

## 九、本地文档与 Hook 规则

### 9.1 本地文档目录

所有文档必须创建在 `answers/` 目录下，路径格式：

```
answers/领域/二级领域/文件名.md
```

### 9.2 Hook 验证规则

1. 顶级目录必须在白名单内（医学、安全、技术、政治、教育科学、法律、游戏、社会科学、管理、经济、自然科学）
2. 路径深度必须是 4 层（`answers/领域/二级领域/文件名.md`）
3. 文件名只允许：中文、字母、数字、`_-().`
4. **第一行必须是标签**：`#xxx` 或 `@xxx`，≥3 个
5. **必须用 `**加粗**` 作标题**，禁止用 `#` `##` 标题
6. 禁止非 flomo 语法：标题、引用、代码块、链接、图片、水平线、表格
7. **标题相似度检测（仅警告不阻断）**：hook 会自动扫描同子目录下（`answers/领域/二级领域/`）已存在的文档，提取文件名关键词做相似度计算。共享 ≥3 个关键词时输出警告提示。警告不阻断 commit，但建议人工确认是否重复后再决定。

**hook 检测到相似度警告时的处理建议**：
- 如果确实是不同角度的新内容：继续 commit
- 如果是同主题的补充：在已有 flomo 笔记上追加，而非新建
- 如果是命名重叠：重命名新文档以区分主题

### 9.3 工作流程

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | `flomo_memo_search` 查重 | **唯一权威查重方式**（必做） |
| 2 | `webfetch` 获取原文 | 不得跳过 |
| 3 | 创建本地文档（flomo 格式） | 第一行标签 + `**加粗**` 标题 |
| 4 | `git add <file>` + `git commit` | hook 自动验证 |
| 5 | 验证失败 | 修复格式重新 add/commit |
| 6 | 验证通过 | `flomo_memo_create` 上传（直接传文档内容） |
| 7 | `git reset --hard HEAD~1` | 清除本地 commit（**永远不要 push**） |

### 9.4 ⚠️ answers 目录严格本地化（重要）

`answers/` 目录是**本地草稿区**，仅用于：
1. hook 格式验证的临时文件
2. 上传 flomo 前的格式测试

**绝对禁止推送到远程**：
- ❌ `git push` 不要包含 answers/ 内容
- ❌ 不要在 answers/ 下存储长期资料
- ✅ 上传 flomo 后立即 `git reset HEAD~1` 清除本地 commit
- ✅ git remote 仓库只保留基础设施：`hooks/`、`scripts/`、`SKILL.md`、`.gitignore` 等
- ✅ `.gitignore` 已包含 `answers/`，新文件不会被 commit
- ✅ 偶尔 commit 的 answers/ 内容（如测试用）必须用 `git reset` 撤回

## 十、操作边界

| 规则 | 说明 |
|------|------|
| **只做本 skill 明确写明的事情** | 未提及的操作一律禁止 |
| **收到 URL 必须 webfetch** | 不得以"之前处理过"为由跳过 |
| **必须 flomo_memo_search 查重** | 创建本地文档前必须调用；hook 的相似度检测只是辅助 |
| **必须创建本地文档** | 禁止直接上传 flomo，必须先创建本地文件并 hook 验证 |
| **answers 严禁推送远程** | 所有 answers/ 文档是临时草稿，上传 flomo 后立即 `git reset HEAD~1` 清除；永远不要 push |
| **更新前必须校验格式** | 更新笔记前必须检查是否符合本文档"二、文档结构"的所有格式规则 |
| 默认产出 | 每条输入都产出笔记，不存在"没有知识价值"的输入 |
| 禁止私自增加限制 | 写入或修改本 skill 时，不得私自增加用户未明确要求的限制 |
| 禁止拒绝产出 | **禁止**以任何理由拒绝生产笔记；抓取不到内容（网络错误、404、登录限制等）除外 |
| 禁止批量操作 | 每条输入单独上传 flomo，完成后再处理下一条 |
