# struct-doc-answer — 知识文档生成

## ⚠️ 强制检查清单

| # | 检查项 | 合格标准 |
|---|--------|----------|
| 1 | 收到 URL → **立即 webfetch** | 不得以"之前处理过"为由跳过 |
| 2 | 已读完整内容 | 不仅是标题/摘要/URL，必须读完全文 |
| 3 | 目录存在性 | 目标目录必须已存在 |
| 4 | 标题格式 | `领域_二级领域_知识点`，三段式，每段≥2字符 |
| 5 | 标签数量 | ≥3个，必含 `#信号类型`，咨询类必含 `@咨询线` |
| 6 | 四章节完整 | `## 来源` `## 概述` `## 核心概念` `## 分类标签` |
| 7 | flomo 上传 | 调用 flomo_memo_create 上传笔记 |
| 8 | 无非标章节 | 禁止 `## 延伸阅读` `## 摘要` `## 引言` `## 结论` `## 思考题` 等 |
| 9 | 分类标签殿后 | `## 分类标签` 必须是文档最后一个章节 |

> **核心红线：禁止跳过。每条输入必须产出文档。禁止使用非标准章节。**

---

## 一、标题与路径规则

### 1.1 标题格式

**格式：** `领域_二级领域_知识点`（名词，三段式）

| 示例 | 文件路径 |
|------|---------|
| `科技_AI芯片_专用处理器设计范式` | `answers/科技/AI芯片/专用处理器设计范式.md` |
| `金融_股票策略_估值模型分析` | `answers/金融/股票策略/估值模型分析.md` |

**知识点三要求：**
- 可独立识别：能用一个名词短语指代一个具体的事件、对象、技术、现象或方法
- 可作为文档主语：文档全部内容围绕这个知识点展开
- 可挂载多个事实：可容纳多个数据点、案例、机制描述

## 二、文档结构

```markdown
# 领域_二级领域_概念标题

## 来源

来源名称（禁止：链接、括号、冒号后跟内容）

## 概述

第一句给出精确定义，第二句交代背景与现实意义，第三句说明核心逻辑、演变脉络或关键争议。

## 核心概念

### 子概念一

先给出精确定义，再解释核心逻辑或运作机制，然后举一个具体事实案例，最后说明该子概念与主概念的关联。

### 子概念二

（同上结构）

## 分类标签

#信号类型 @咨询线 #领域 #二级领域 #具体概念

**上传 flomo 时，将标签移到最前面，格式：**

```markdown
#信号类型 @咨询线 #领域 #二级领域 #具体概念

**领域_二级领域_概念标题**

来源：来源名称

概述：...

核心概念：

- ...
```

**flomo MCP 工具：**
- `flomo_memo_create`：上传笔记到 flomo
- `flomo_memo_search`：在 flomo 中搜索已有笔记
**来源格式规范：**
- 只保留来源名称，去除URL和冗余描述
- 禁止使用括号、链接、冒号后跟内容
- 格式：`中国新闻网，新华社`（两个来源用中文顿号分隔）
- 禁止示例：`来源：XX（网址）`、`36氪快讯：正文内容`、`来源：XXhttp://xxx`

**标签顺序（从左到右）：**
1. `#信号类型`（必选，五选一）
2. `@咨询线`（可选，咨询类文档必选）
3. `领域`（必选，来自标题的"领域"层级，**带#号**，如 `#技术`）
4. `二级领域`（必选，来自标题的"二级领域"层级，**带#号**，如 `#AI大模型`）
5. `具体概念`（可选，描述文档核心概念，**带#号**，数量不限）

**示例：**
- 标题：`科技_AI大模型_Grok4.5开放` → 标签：`#信号笔记 @竞争情报 @行业研究 #科技 #AI大模型`
- 标题：`政治法律_马克思主义_智能机器人价值论商榷` → 标签：`#知识基座 #政治法律 #马克思主义 #劳动价值论 #政治经济学`
- 标题：`经济_农业经济_数字乡村农业全要素生产率增长` → 标签：`#趋势信号 #经济 #农业经济 #数字乡村 #农业TFP #规模经营`

**强制规则：**
- 只能包含上述 4 个标准章节
- **禁止**使用以下非标准章节：`## 基本信息` `## 概念解释` `## 相关案例` `## 拓展思考` `## 起源` `## 参考文献` `## 元信息` `## 延伸阅读` `## 摘要` `## 引言` `## 结论` `## 开头` `## 正文` `## 结语` `## 思考题` `## 应用场景`
- **来源格式**：标准格式用 `## 来源`，flomo 格式用 `来源：`
- **标签格式**：标签横写，flomo 格式标签放在正文最前面
- **分类标签横写**：标签在一行内连续书写，禁止换行或列表格式
- **核心概念**：flomo 格式用 `-` 列表，标准格式用 `### 子概念名`

- **禁止链接格式**：来源中禁止使用 `[标题](链接)` 格式

## 三、核心操作流程

### 3.1 标准流程（网页/文章）

| 步骤 | 操作 | 校验点 |
|------|------|--------|
| 1 | `webfetch <URL>` | 获取完整内容 |
| 2 | 确定标题 `领域_二级领域_知识点` | 标题格式正确？三级结构？ |
| 3 | 分配标签 | ≥3个？含 `#信号类型`？咨询类含 `@咨询线`？ |
| 4 | 按 flomo 模板生成内容 | 标签在最前，段落空行 |
| 5 | `flomo_memo_create` 上传 | 返回 id 和 url |

### 3.2 微信公众号流程

1. `webfetch` 获取正文（format=html）
2. 解析HTML提取正文文本内容
3. 按 flomo 模板生成内容并上传

### 3.3 期刊目录页流程

**必须逐一处理每篇论文，禁止合并。**

1. 识别目录页中所有论文
2. 逐篇处理：每篇独立标题 → 生成内容 → flomo 上传
3. flomo 上传必须逐篇进行，不得合并

## 四、标签体系

### 4.1 咨询服务线标签（@前缀）

咨询类文档强制，至少1个，通常不超过3个

| 标签 | 适用场景 |
|------|---------|
| `@行业研究` | 行业深度分析、市场规模评估、竞争格局梳理 |
| `@战略规划` | 客户战略制定、增长策略、业务组合优化 |
| `@竞争情报` | 竞争对手动态追踪、竞品分析 |
| `@政策预警` | 监管政策变化评估、合规风险识别 |
| `@运营诊断` | 供应链/渠道/效率分析 |
| `@市场进入` | 新市场/新客户进入策略 |
| `@技术尽调` | 技术评估，投资标的分析 |
| `@风险管理` | 地缘政治/供应链/合规风险评估 |
| `@组织人力` | 组织变革、人才策略、薪酬设计 |
| `@资本咨询` | 资本运作、并购、IPO策略 |

### 4.2 信号类型标签（#前缀）

每个文档必须选择一个

| 标签 | 含义 | 特征 | 时效性 |
|------|------|------|--------|
| `#知识基座` | 长期稳定的概念/机制/框架 | 体系化论述，含机制解释和多案例 | 数年有效 |
| `#趋势信号` | 正在发生的结构性变化 | 含趋势判断、方向性分析 | 季度-年度有效 |
| `#信号笔记` | 单次事件/数据点记录 | 以事件为核心，含咨询含义推断 | 周-月有效 |
| `#分析框架` | 可复用的方法论/分析模型 | 非领域知识，是思考工具 | 数年有效 |
| `#知识载体` | 工具/资源/数据集/可视化 | 承载和呈现结构化知识 | 数年有效 |

> 企业/行业/监管信息默认 `#信号笔记`

### 4.3 标签格式规范

**所有标签必须以 `#` 或 `@` 开头，无一例外。**

| 检查项 | 标准 |
|--------|------|
| 数量 | ≥3个 |
| 必含 | `#信号类型`（五选一） |
| 咨询类 | 必含 `@咨询线`（至少1个） |
| 领域 | 标题中的一级领域，**必须加 `#`**，如 `#技术` `#政治` |
| 二级领域 | 标题中的二级领域，**必须加 `#`**，如 `#AI大模型` `#外交` |
| 格式 | 标签之间用空格分隔，无其他符号包围 |

**正确示例：**
```
## 分类标签
#信号笔记 #技术 #AI大模型
```

**错误示例（漏写 #）：**
```
## 分类标签
#信号笔记 技术 AI大模型  （错误：技术和AI大模型没有#前缀，不被识别为标签）
```

**完整标题示例：**
- 标题：`教育科学_科技教育_青少年机器人设计与信息素养大赛`
- 标签：`#信号笔记 #教育科学 #科技教育 #机器人竞赛 #信息素养`

## 五、同一概念判断

| 场景 | 判断 | 操作 |
|------|------|------|
| 核心术语完全相同 | ✅ 同一概念 | 更新补充新角度/案例 |
| 术语不同但互为同义词或上下位关系 | ✅ 同一概念 | 合并到更宽泛或更成熟的文件 |
| 同一事件/政策的不司角度分析 | ✅ 同一概念 | 合并到同一文件的不同子概念 |
| 核心术语相同但分析视角完全不同 | ⚠️ 视情况 | 若子概念可容纳则合并，否则新建并互链 |
| 不同领域、不同实体、不同分析框架 | ❌ 不同概念 | 新建独立文件 |

## 六、更新规则

### 6.1 更新原则：增量修改，禁止覆盖

**判断标准：**
- 标题中的核心术语与已有文档完全相同 → 增量更新
- 标题中的核心术语部分重叠、可以合并 → 增量更新
- 仅来源/案例/数据不同，概念相同 → 增量更新

**正确更新方式：**

| 要更新的内容 | 操作方法 |
|------------|---------|
| 新来源 | 追加到 `## 来源` 已有来源的**同一行末尾**，用顿号分隔，去重 |
| 新子概念/新机制/新技术 | 在 `## 核心概念` 中插入新的 `### 子概念名` 小节 |
| 新事实/新案例 | 追加到已有 `### 子概念名` 小节的**末尾段落** |
| 标签补充 | 替换 `## 分类标签` 行，保留原有标签，新增补充的 |

**禁止全量覆盖：**
- ❌ 禁止用 `write` 直接重写整个文件
- ❌ 禁止删除原文重新创建
- ❌ 禁止用新内容替换旧内容的方式"更新"

**禁止行为：**
- ❌ **禁止**在 `## 分类标签` 之后插入任何内容
- ❌ **禁止**在 `## 来源` 之后、`## 概述` 之前插入内容
- ❌ **禁止**用新章节（如 `## 延伸阅读`）收纳内容

## 七、咨询视角信号分析法

### 第一层：存在性追问
- 为什么是这个时间点出现？
- 为什么是这个公司/这个人？
- 为什么这个故事被报道？（主动释放信号/被动回应猜测/行业吹风）

### 第二层：咨询含义推断
- 这篇文章对什么类型的咨询问题有用？
- 这是趋势信号还是噪音？
- 是否需要与其他文档交叉印证？

### 第三层：文档类型选择
- `#知识基座` → 创建/更新机制文档
- `#趋势信号` → 创建/更新趋势判断文档
- `#信号笔记` → 创建事件记录文档

## 八、旧文档格式转换

| 旧格式 | 替换为 |
|--------|--------|
| `## 摘要` `## 引言` `## 结论` | 合并到 `## 概述` |
| `## 理论基础` `## 分析与论证` | 合并到 `## 核心概念`（`###`子节） |
| `## 概念解析` `## 新闻例证` | 分别合并到 `## 概述` 和 `## 核心概念` |
| `## 开头` `## 正文` `## 结语` | 重排到 `## 概述`、`## 核心概念` |
| `## 拓展思考` `## 起源` `## 参考文献` `## 延伸阅读` | 删除（内容有价值的合并到相关章节） |
| `## 元信息`（表格） | 合并为一行 `## 来源` |

## 九、客户语境笔记

**路径格式：** `answers/一级分类/_clients/客户名_知识点标识.md`

**必须回答三个问题：**
1. 发生了什么（引用原始知识文档标题）
2. 对客户X意味着什么（直接判断）
3. 下一步建议（可操作动作）

> 可过期归档至 `_clients/_archived/`

## 十、操作边界

| 规则 | 说明 |
|------|------|
| **只做本skill明确写明的事情** | 未提及的操作一律禁止 |
| **收到 URL 必须 webfetch** | 不得以"之前处理过"为由跳过 |
| **更新前必须校验格式** | 更新笔记前必须检查是否符合本文档"二、文档结构"的所有格式规则 |
| 默认产出 | 每条输入都产出笔记，不存在"没有知识价值"的输入 |
| 禁止私自增加限制 | 写入或修改本 skill 时，不得私自增加用户未明确要求的限制 |
| 禁止拒绝产出 | **禁止**以任何理由拒绝生产笔记；抓取不到内容（网络错误、404、登录限制等）除外 |
| 禁止批量操作 | 每条输入单独上传 flomo，完成后再处理下一条 |
